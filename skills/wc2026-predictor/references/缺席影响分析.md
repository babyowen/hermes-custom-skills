# 关键缺阵影响分析机制

## 概述

当一名球员被确定缺席世界杯（❌）时，每次采集 cron 需自动触发影响分析：分析该球员缺阵对该队战术、阵容深度、小组出线形势的影响，并记录到知识库中。

## 触发条件

| 触发场景 | 说明 |
|---------|------|
| 伤病追踪 🏥 | 球员状态从 `⚠️`→`❌`，或新发现某球员确认缺席 |
| 落选追踪 📋 | 知名球员意外落选最终26人名单 |
| 新增来源 | 首次确认（通过BBC/ESPN/SI/Reuters等可信来源） |

## 分析框架（5维度）

对每个确认缺席的球员，按以下5个维度进行分析：

### ① 球员角色
| 角色 | 含义 | 
|------|------|
| 🏆 绝对核心 | 球队战术核心，缺阵导致体系崩溃或重大重构 |
| ⭐ 主力首发 | 稳定首发11人，有明确战术职责 |
| 🔄 关键轮换 | 主力-替补之间的重要轮换，大赛中通常累计2-3场首发 |
| 🧩 功能性替补 | 特定战术/阵型下的专才替补（如定位球战术、防守型中场） |
| 📋 潜力新星 | 经验有限但被看好能在大赛爆发的年轻球员 |

### ② 损失分析
定性和定量描述缺阵带来的损失：
- **进攻端**: 进球贡献、助攻、突破/创造力、定位球能力
- **中场端**: 控制力、向前传球、防守覆盖、节奏控制
- **防守端**: 单防能力、指挥防线、空中优势、回追速度
- **精神属性**: 领袖气质、大赛经验、更衣室影响
- **战术灵活性**: 可打多个位置、特定战术专属执行者

### ③ 替代方案评估
| 维度 | 评分 |
|------|:----:|
| 同位置替代人选 | 有/有但档次差距大/无 |
| 替代者水平 | 一流/合格/档次降级/无人可用 |
| 战术调整 | 无需变阵/微调/大幅重构 |
| 整体实力影响 | 致命打击（掉0.5档）/ 显著削弱（掉0.3档）/ 可控损失（掉0.1档） |

### ④ 战术影响
- 阵型是否需要调整
- 进攻套路/防守体系是否受影响
- 教练是否因此被迫改变计划（如多带一名同位置球员）
- 对其他位置的连锁反应（如某人被推到不熟悉的位置）

### ⑤ 出线形势影响
- **小组赛**: 该队出线概率变化（↑/→/↓）
- **淘汰赛**: 对后续潜在对手的威胁程度变化
- **特别关注**: 伤在该队没深度位置的致命性 > 伤在人才济济的位置

### 位置1: `concepts/关键缺阵影响分析.md`（总表）
```

| 维度 | 分析 |
|------|------|
| 球员角色 | 🏆 绝对核心 / ⭐ 主力首发 等 |
| 损失分析 | 进攻/防守/精神方面损失 |
| 替代方案 | 替代者 + 实力降级评估 |
| 战术影响 | 阵型/战术变化 |
| 出线影响 | 概率变化 + 特别关注 |

```

## 执行流程

每次采集 cron 中：

```
Step 1: 读取伤病追踪总表中的 ❌ 标记球员
Step 2: 对比上次运行记录，识别新增的 ❌ 确定缺席
Step 3: 对每个新增 ❌ 球员，用 web_search 获取最新背景信息
Step 4: 按5维度框架完成影响分析
Step 5: 写入 concepts/关键缺阵影响分析.md
Step 6: 同步到对应 entities/{队名}.md
Step 7: 同步到该队全部小组赛比赛页的 伤病动态 节
Step 8: 在 Feishu 报告中注明新增 🆕 缺阵影响分析
```

## 去重与更新规则

| 规则 | 说明 |
|------|------|
| 首次确认 | 创建完整5维度分析，标记 `首发日期: YYYY-MM-DD` |
| 信息更新 | 球员已有分析记录但获取到新信息 → 更新 `最新更新: YYYY-MM-DD` |
| 重复不记 | 同一球员同一信息多次提及 → 不重复分析，只更新日期 |
| 批量处理 | 同一次运行中多个确认缺席 → 分别分析，并行写入 |

## 信息来源优先级

| 优先级 | 来源 | 说明 |
|:-----:|------|------|
| 1 | 官方公告 | 球队/FIFA/主教练确认 | 
| 2 | BBC/ESPN/SI/Reuters | 主流体育媒体独立报道 |
| 3 | 当地媒体 | 该球员所属国家/联赛的当地报道 |
| 4 | SerpAPI摘要 | 当全文提取失败时，从搜索结果摘要获取足够信息 |

## 写入安全规则

⚠️ **安全关键**: 写入 `concepts/关键缺阵影响分析.md` 时，永远使用 `execute_code` 中的 `open(path, 'a')` 追加模式，或 `terminal` 中的 `cat >>` 操作。**绝不能用 `read_file` + `write_file` 组合**（read_file输出含行号前缀，写入会污染文件）。参见 wiki-ingestion-pipeline 的 `read_file output -> write_file` 陷阱。
